# 背屏录屏声音录制踩坑记录

> **Shizuku / 存储 / shell 启动**：见 [Shizuku权限注意事项.md](./Shizuku权限注意事项.md)。

## 核心原则

**永远不要使用 `screenrecord --audio` 内录捷径，统一走 MediaProjection + AudioCaptureHelper (PCM) 路径。**

---

## 1. `screenrecord --audio` 不可靠

`screenrecord --audio` 是 Android 12 (API 31) 引入的功能，但**部分 ROM（含 HyperOS）不支持或行为异常**：

- `screenrecord --help` 可能**列出** `--audio` 但实际运行时静默失败（无声）
- 参数名因 ROM 而异：AOSP 是 `--audio-bitrate`，但某些 ROM 可能不认
- 一旦 `--audio` 失败，整段录屏可能直接无音轨，无法补救

**结论**：不要在录制阶段依赖 `--audio`。screenrecord **只录画面**，音频单独采 PCM。

---

## 2. MediaProjection + PCM 是唯一可靠路径

参照旧版 (3.4) 方案，始终：

1. 用户开启「录音」开关 → 请求 `MediaProjection` 授权（`MediaProjectionRequestActivity`）
2. screenrecord 启动时**不带任何 `--audio` 参数**，只录画面
3. `AudioCaptureHelper` 通过 `AudioPlaybackCapture` API 独立采集 PCM 音频
4. 停止后通过 **Media3 Transformer** 将 PCM 合并入 MP4

关键代码位置：
- 录制入口：`RearScreenRecordService.kt` → `startRecordingInternal()`
- 音频采集：`AudioCaptureHelper.kt`
- 音视频合并：`ShellMedia3Export.kt` → `mergePcmIntoMp4()` / `mergePcmThenCompositeShell()`

---

## 3. AudioCaptureHelper 采样率/声道顺序

这是最容易踩的坑。**不同 App 对采集参数的兼容性不同**。

### 酷我车载 (`cn.kuwo.kwmusiccar`) 的问题

- 48kHz Stereo 优先时 → 其他 App 正常，酷我**无声**
- 44.1kHz Mono 优先时 → 全部正常

### 正确的 profile 顺序

```kotlin
// ✅ 正确：44.1kHz Mono 优先（兼容性最好）
val profileOrder = listOf(
    CaptureProfile(sampleRate = 44100, channelMask = CHANNEL_IN_MONO, channelCount = 1),
    CaptureProfile(sampleRate = 44100, channelMask = CHANNEL_IN_STEREO, channelCount = 2),
    CaptureProfile(sampleRate = 48000, channelMask = CHANNEL_IN_MONO, channelCount = 1),
    CaptureProfile(sampleRate = 48000, channelMask = CHANNEL_IN_STEREO, channelCount = 2),
)
```

```kotlin
// ❌ 错误：48kHz Stereo 优先 → 酷我无声
val profileOrder = listOf(
    CaptureProfile(sampleRate = 48000, channelMask = CHANNEL_IN_STEREO, ...),
    ...
)
```

### 原因分析

`AudioRecord` 以某个 profile 成功初始化（`STATE_INITIALIZED`），不代表能正确采到所有 App 的音频。部分 ROM 在 48kHz Stereo 配置下对特定 App（如车载音乐 App）的音频路由存在兼容问题，表现为：
- `AudioRecord.read()` 返回正常字节数（无报错）
- 但 PCM 数据实际为静音或噪声
- 最终合并出的视频无声音

**优先 44.1kHz Mono（最广泛兼容的配置）可避免此问题。**

---

## 4. 不要显示「系统内录不可用」

旧代码在 `screenrecord --audio` 不可用时弹出 Toast "系统内录不可用"，然后回退到 PCM 路径。这条 Toast 误导用户以为内录完全不可用，但实际上 PCM 路径仍然正常工作。

**正确做法**：不探测 `--audio` 支持，不显示此提示，直接走 PCM。

---

## 5. 变更检查清单

修改背屏录屏声音相关代码时，确认以下各项：

- [ ] 没有 `screenrecord --audio` 或 `--audio-bitrate` / `--audio-bit-rate` 参数
- [ ] 没有 `screenrecordSupportsAudio()` 探测调用
- [ ] `AudioCaptureHelper` profile 顺序为 44.1kHz Mono 优先
- [ ] 录制入口直接走 `requestMediaProjectionThenStart()`，不先探测内置音频
- [ ] 不存在 `sessionAudioBuiltin` 字段及相关分支
- [ ] 没有 `compositeShellOverVideoWithCallback()` 死代码
- [ ] 没有 "系统内录不可用" (`record_audio_builtin_unavailable`) Toast

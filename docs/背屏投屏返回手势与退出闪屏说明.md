# 背屏投屏：系统返回手势受限与退出闪白屏说明

本文基于 **MiRoot 1.8.4** 音乐投屏、车控投屏与背屏桌面对齐后的实现整理，说明两类问题的**成因**与**解法**。充电动画单独策略见 [充电动画判断、迁移策略与窗口属性说明.md](./充电动画判断、迁移策略与窗口属性说明.md)。

| 文件 | 职责 |
|------|------|
| `rear/RearMirootProjectionLifecycle.kt` | 背屏不透明窗口、系统返回就绪、`finish` 前唤醒、延后恢复官方背屏 |
| `RearDisplayInputHelper.java` | 清除 `NOT_TOUCHABLE` / `NOT_FOCUSABLE` 等，保证窗口可触摸、可聚焦 |
| `rear/RearProjectionLaunchSequence.kt` | 背屏直启优先 → 失败则主屏占位迁屏；启动时 `OfficialSubscreenMiRootProjectionSession.acquire` |
| `rear/OfficialSubscreenMiRootProjectionSession.kt` | 引用计数禁用/恢复官方背屏中心（`disableSubScreenLauncher`） |
| `lyrics/RearScreenLyricsActivity.java` | 音乐投屏 UI、退出、迁屏轮询 |
| `car/RearScreenCarControlActivity.java` | 车控投屏 UI、退出、迁屏轮询 |
| `rear/desktop/RearScreenDesktopActivity.kt` | 背屏桌面（行为对齐参考） |
| `charging/RearScreenChargingActivity.java` | 充电动画（退出语义对齐参考） |

---

## 1. 问题现象

| 现象 | 典型场景 |
|------|----------|
| **系统边缘返回长时间无效** | 手势/广播启动音乐或车控后，背屏已出画面，但侧滑/系统返回要等数秒才生效；背屏桌面则「一启动就能返回结束」 |
| **结束投屏闪白屏** | 底部滑退、返回键、通知结束等路径下，背屏先白一下再黑或再切到官方 Launcher |
| **结束投屏长时间黑屏** | 退出时先压黑、`finishAndRemoveTask` 或在 `finish` 前恢复官方背屏，背屏空窗过久 |

---

## 2. 系统返回手势受限

### 2.1 背屏桌面为何「启动即可返回」

背屏桌面 `RearScreenDesktopActivity` 启动链路特点：

1. **`am start --display 1` 直启**：Activity 从第一帧就在背屏（`displayId == 1`），`onCreate` 内即 `setContent` + `applyRearWindowFlags()`。
2. **生命周期里持续清输入限制**：`onResume` / `onWindowFocusChanged` 每次调用 `RearDisplayInputHelper.ensureApplicationWindowReceivesInput()`。
3. **沉浸式只藏状态栏**：`hideSystemUi()` 使用 `BEHAVIOR_DEFAULT` 且 **保留 navigationBars / 手势 inset**，边缘返回不与「拉出系统栏」争用。
4. **启动时已禁用官方背屏**：`RearProjectionLaunchSequence.finalizeOnRear()` → `OfficialSubscreenMiRootProjectionSession.acquire()`，边缘手势由当前 App 窗口接收，而非 `com.xiaomi.subscreencenter`。

### 2.2 音乐 / 车控为何曾「启动不能立刻返回」

音乐、车控与桌面共用「背屏直启优先，失败则主屏占位迁屏」，但 Activity 侧曾有多处差异：

#### （1）主屏占位阶段窗口在 display 0

迁屏前 Activity 跑在主屏透明占位上，**用户看的背屏**与 **窗口所在 Display** 不一致，系统边缘返回落在背屏时，事件到不了仍挂在主屏的 MiRoot 窗口。

#### （2）`FLAG_NOT_TOUCHABLE` 未及时清除

退出路径、停止 Intent、非法表面销毁等曾设置 `FLAG_NOT_TOUCHABLE`；迁屏或 `onResume` 若未清标志，背屏窗口长期不可触摸，**系统手势也无法送达**。

`RearDisplayInputHelper` 注释已说明：

> 结束投屏等路径会临时加 `FLAG_NOT_TOUCHABLE`；正常展示副屏时应清除，避免「背屏完全不响应手势」。

#### （3）车控 `OnBackPressedCallback` 注册过晚

车控曾在 `createUI()` 末尾才 `registerCarControlBackPressedCallback()`。占位迁屏后 UI 未 inflate 前，系统返回无回调承接。歌词虽在 `onCreate` 注册，但若窗口不可触摸/无焦点，同样无效。

#### （4）背屏 UI 初始化等待过长

占位迁屏后若 `onResume` **固定延迟**（例如 2.5s）才建 UI，或仅靠单次 `onResume` 判断，窗口在背屏已显示数秒内仍无内容、无焦点。

#### （5）官方背屏中心与 TaskService 绑定偏晚

`disableSubScreenLauncher` 在投屏启动时由 Session 已调用一次，但 Activity 内 `bindTaskService` 连接前若官方进程仍接管背屏边缘手势，返回会进官方 Launcher 而非结束投屏。广播路径还曾依赖 `ProjectionHelper` 后台线程等待 UI + TaskService（数百毫秒～数秒）再反射调用屏蔽。

#### （6）沉浸式误伤手势区（历史问题）

若对背屏使用 `IMMERSIVE_STICKY` 并隐藏导航栏，系统会把边缘滑动当成「拉出系统栏」，而不是应用内返回。当前音乐/车控已与桌面一致：**仅隐藏状态栏，保留导航栏与手势 inset**。

### 2.3 解决方法（当前实现）

#### 共享：`RearMirootProjectionLifecycle.primeRearSystemBackGestures()`

在 **背屏 `displayId == 1`** 时：

1. `RearDisplayInputHelper.ensureApplicationWindowReceivesInput()` — 清除不可触摸/不可聚焦标志；
2. `decorView.requestFocus()` — 让系统返回能投递到当前 Activity。

调用时机：`onCreate`（直启背屏）、迁屏 **40ms 轮询** 检测到背屏、 `initXxxUiOnRearIfNeeded`、`onStart` / `onResume` / `onWindowFocusChanged`（获焦且背屏）。

#### 迁屏轮询（40ms × 最多 45 次）

主屏占位 `onCreate` 后启动 `schedulePollRearLyricsUiInitAfterMove` / `schedulePollRearUiInitAfterMove`：一旦 `displayId == 1` 先 `primeRearSystemBackGestures`，再建 UI，避免「已到背屏仍盲等」。

常量：`RearMirootProjectionLifecycle.REAR_UI_INIT_POLL_INTERVAL_MS`、`REAR_UI_INIT_POLL_MAX_ATTEMPTS`。

#### 尽早注册返回回调

- **车控**：`onCreate` 开头即 `registerCarControlBackPressedCallback()`（与歌词一致），`finishProjectionFromUser()` 处理系统返回/滑退/通知。
- **歌词**：保持 `onCreate` 注册 `registerLyricsProjectionBackPressedCallback()`。

#### 占位阶段也清标志、预热 TaskService

- 主屏占位 `onCreate` / `onResume`：`ensureApplicationWindowReceivesInput()`（不要求 display 1，为迁屏后衔接做准备）。
- `RootTaskServiceConnector.prewarm()`：`onCreate` 预热，减少冷绑延迟。
- `TaskService.onServiceConnected`：立即 `checkAndDisableOfficialGesture()` + `RearMirootProjectionLifecycle.reinforceOfficialSubscreenDisabled()`；`disableOfficialGesture()` 可回退 `RootTaskServiceConnector.getIfConnected()`。

#### 车控占位保留实例

主屏占位时设置 `currentInstance = this`，便于启动链路与轮询识别 Activity，避免「实例为空、手势巩固逻辑跳过」。

#### 与背屏桌面对齐的沉浸式

`hideSystemUI()`：`WindowInsetsController.show(navigationBars)` + `hide(statusBars)` + `BEHAVIOR_DEFAULT`。

### 2.4 背屏返回偶发露出 MiRoot 主界面（MainActivity）

**现象**：占位迁屏后，在背屏系统返回结束投屏，**主屏**偶发出现 MiRoot **应用主界面**（Flutter 主页），而不是直接回到系统桌面或仅结束背屏。

**原因**（与「主屏闪歌词/车控 UI」不同）：

- 音乐/车控投屏使用**独立** `taskAffinity`，与 `MainActivity` 不在同一 task。
- 用户此前打开过 MiRoot 时，**主屏**任务栈顶往往是 `MainActivity`。
- 背屏 `finish()` / `finishAndRemoveTask()` 移除投屏 task 后，**主屏焦点回落**到仍在栈内的 `MainActivity`，并非投屏 Activity 在主屏又画了一遍 UI。

**解法**：`RearMirootProjectionLifecycle.sendMainDisplayHomeBeforeProjectionEnd()` — 在 MiRoot 主动结束投屏的 `finish()` 中对 **display 0** 执行一次 `KEYCODE_HOME` + `am start … HOME`（800ms 内去重；`onDestroy` 恢复官方背屏不再重复 HOME），先把主屏切回系统桌面。

调用点：音乐/车控 `finishProjectionFromUser` 与 `finish()`；充电 `finishRequestedByMiRoot`；`scheduleOfficialSubscreenRestoreAfterDestroy` 线程首步。

### 2.5 主屏占位迁屏：背屏返回偶发主屏露出投屏 UI

**现象**：走主屏占位 + `moveTaskToDisplay` 时，在背屏做系统返回，偶发**主屏**闪一下歌词/车控界面，投屏未立刻结束。

**原因**：

1. `singleInstance` + 独立 `taskAffinity` 的 Activity 在背屏 `finish()` 时，系统可能先把 **task 落回 display 0** 再销毁。
2. 此时 `onResume` 在主屏执行，且 `lyricsView` / `mainLayout` 已存在，旧逻辑会走「UI 已创建，更新常亮 flags」等分支，**在主屏绘制了一帧投屏 UI**。
3. 用户结束若仅用 `finish()` 不用 `finishAndRemoveTask()`，task 在主屏残留片刻，更容易被看到。

**解法**（`RearMirootProjectionLifecycle` + 音乐/车控 Activity）：

| 措施 | 说明 |
|------|------|
| `hideWindowBeforeProjectionFinish()` | `finishProjectionFromUser` / `finish()` 前将 decor、`content` 设为 `GONE` |
| `shouldSkipProjectionResume()` | `isFinishing` / 退出流程中不再 `onResume` 刷新 UI |
| `shouldFinishOnMainDisplayDuringProjection()` | `initialDisplayId==1` 且当前在主屏且已有投屏 UI → 立即 `finishProjectionFromUser` |
| 用户结束且 `initialDisplayId==1` | `finish()` 内使用 `finishAndRemoveTask()`，避免 task 挂到主屏（仍 `onDestroy` 后延迟 restore 官方背屏，避免闪白） |

### 2.6 主屏禁止展示投屏 UI（仅透明占位）

**要求**：走主屏占位 + 迁屏的 **音乐 / 车控 / 背屏桌面 / 充电动画**，在主屏（`displayId == 0`）上：

- **不得** `setContentView` / Compose `setContent` / 绘制黑底投屏界面；
- **仅** 透明主题窗口占位，等待 `moveTaskToDisplay` 或直启到背屏；
- **即便迁屏失败**、task 落回主屏，也**不能**在主屏露出歌词/车控/桌面/充电视图；若已有 UI 则隐藏并结束投屏。

**例外**：音乐投屏 **主屏横屏磁贴**（`isMainScreenLandscapeMode`）允许在主屏展示 UI。

**实现**（`RearMirootProjectionLifecycle`）：

| API | 作用 |
|-----|------|
| `resolveMainDisplayProjectionMode()` | 返回 `TRANSPARENT_PLACEHOLDER` / `MUST_END_PROJECTION` / `NOT_APPLICABLE` |
| `applyMainDisplayTransparentPlaceholder()` | `PixelFormat.TRANSLUCENT` + 透明 `windowBackground` |
| `applyMainDisplayPlaceholderPolicy()` | 按 mode 应用；`MUST_END` 时额外 `hideWindowBeforeProjectionFinish()` |

各 Activity 在 `onCreate` / `onResume` / `onConfigurationChanged` 落主屏时先解析 mode，占位则 `return`，已有 UI 则 `finishProjectionFromUser` / `finishFromMiRoot` / `finish()`。

### 2.7 自测要点

1. 手势/广播启动音乐、车控 → 背屏**刚有画面**（UI 可仍在加载）即试系统边缘返回，应能结束投屏。
2. **刻意走占位迁屏**（关闭直启或直启失败）→ 背屏系统返回，主屏**不应**闪歌词/车控，应直接结束。
3. 对比背屏桌面：两者均应在禁用官方背屏中心后，由 MiRoot 窗口承接返回。
4. 若仅「占位迁屏」慢、直启快，查 `RearProjectionLaunchSequence` 直启是否验证失败回退占位。

## 3. 退出闪白屏与长时间黑屏

### 3.1 闪白屏成因

#### （1）透明主题 + 浅色内容

Manifest 使用 `Theme.MiRoot.Transparent` 时，窗口默认透明；退出瞬间若内容层收起，会透出 **系统日间白底** 或官方 Launcher 浅色界面 → 用户感知为「闪白」。

#### （2）在 `finish()` 之前 `Session.release` / 恢复官方 Launcher

`OfficialSubscreenMiRootProjectionSession.release()` 会 `enableSubScreenLauncher()`，**在 Activity 仍可见时** 拉起官方背屏 Launcher（常为白底），与 MiRoot 窗口叠加以产生白闪。

背屏桌面注释（`RearScreenDesktopActivity.onDestroy`）已强调：从桌面 `am start` 其它应用到背屏时 **勿在 onDestroy 随意 release**，否则会抢占刚启动的应用。

#### （3）退出前「压黑 + NOT_TOUCHABLE」

旧逻辑在 `finish` 前 `blackoutWindowForProjectionExitImmediate()`、设黑底并 `FLAG_NOT_TOUCHABLE`。`cleanupWindow()` 若清掉 `NOT_TOUCHABLE`，下一帧又可能露出透明区或白底，且与系统转场叠加重闪。

#### （4）`cleanupWindow(true)` / `setDecorFitsSystemWindows(true)` 在 finish 前执行

透明主题下改变 decor insets，会露出原先被遮住的区域，表现为闪白（歌词 `cleanupWindow` 注释已记录）。

### 3.2 长时间黑屏成因

1. **`finishAndRemoveTask()`**：清 task 较慢，背屏在 MiRoot 销毁与官方界面出现之间 **空窗偏长**。
2. **`finish` 前同步重清理**：阻塞 UI 线程，黑屏停留可感知。
3. **`onDestroy` 再次压黑或改透明**：与「并行恢复官方背屏」叠加，黑屏拖长（车控曾有多轮迭代）。

### 3.3 对齐基准：充电动画 / 背屏桌面

**充电动画**（`RearScreenChargingActivity`）：

- 用户结束：`finishRequestedByMiRoot = true` → `finish()`；
- `finish()` 内仅 `prepareRearProjectionVisibleBeforeFinish()`（`KEYCODE_WAKEUP`），**不**在 finish 前 `Session.release`；
- `onDestroy` 后后台线程 `sleep(50ms)` 再恢复官方背屏或下层投屏 task。

**背屏桌面**：

- 滑退/结束：黑底 + `finish()`，**不** finish 前全量 cleanup；
- 官方背屏恢复由 `RearSwitchKeeperService` 等统一收口，避免 onDestroy 误 release。

### 3.4 解决方法（当前实现）

#### 共享：`RearMirootProjectionLifecycle`

| 方法 | 作用 |
|------|------|
| `applyRearOpaqueWindowBase()` | `PixelFormat.OPAQUE` + 不透明 ColorDrawable（黑/车控夜间色），消除透明底 |
| `prepareRearDisplayBeforeFinish()` | `finish()` 前仅背屏 `KEYCODE_WAKEUP` |
| `scheduleOfficialSubscreenRestoreAfterDestroy()` | `super.onDestroy()` 之后延迟 **50ms** 再 `OfficialSubscreenMiRootProjectionSession.release()` |

#### 用户主动结束统一入口

- **音乐**：`finishProjectionFromUser(reason)` → 取消通知 → `finish()`；资源清理在 `onDestroy` 的 `performProjectionResourceCleanup(!finishRequestedByMiRoot)`。
- **车控**：`finishProjectionFromUser(reason)`，逻辑同上。

**禁止**在用户结束路径上：

- `blackoutWindowForProjectionExitImmediate()`；
- `finish` 前 `performCleanupAndExit()` / `Session.release()`；
- 用户结束使用 `finishAndRemoveTask()`（仅异常/门禁路径可保留）。

#### `finish()` 重写

```text
if (finishRequestedByMiRoot) prepareRearDisplayBeforeFinish(displayId, taskService);
super.finish();
overridePendingTransition(0, 0);
```

#### `onDestroy` 顺序

```text
performProjectionResourceCleanup(restoreOfficial = !finishRequestedByMiRoot);
super.onDestroy();
if (finishRequestedByMiRoot) scheduleOfficialSubscreenRestoreAfterDestroy(...);
```

- 用户结束：`restoreOfficial = false`，不在销毁前拉起官方 Launcher。
- 异常/非法表面销毁：`restoreOfficial = true`，仍可按需同步恢复。

#### 沉浸式与退出一致

退出时不改 `setDecorFitsSystemWindows(true)`；用户结束路径不跑 `cleanupWindow` 清触摸标志后再闪一帧。

### 3.5 自测要点

1. 返回 / 底部滑退 / 通知结束：背屏**无白闪**、**无长时间纯黑**。
2. 与充电动画交替：结束充电后下层音乐投屏仍正常，无白闪回归。
3. 异常销毁（占位超时、非法 display）：允许恢复官方背屏，可接受短闪但不应卡黑数秒。

## 4. 启动链路与延迟（辅助说明）

系统返回「慢」常与 **启动慢** 混淆，需区分：

| 环节 | 说明 |
|------|------|
| TaskService 冷绑 | 已用 `RootTaskServiceConnector` + Application/手势 `prewarm`，轮询 30ms |
| 投屏后同步 `Thread.sleep` | `ProjectionHelper` 经 Service 启动时改后台 `scheduleDeferredPostLaunchSteps`，不阻塞广播 |
| 占位迁屏 | `RearProjectionLaunchSequence` 直启验证失败才回退；迁屏 settle 约 40ms + 短轮询确认 |
| postLaunch 唤醒 | `PLACEHOLDER_POST_WAKE_SETTLE_CAP_MS` 等上限，避免按常亮间隔长时间 sleep |

**返回手势**与 **启动耗时** 独立：即使启动仍需 1～2s 出完整 UI，背屏落屏后应能立即边缘返回结束（本文件第 2 节）。

## 5. 新增背屏投屏 Activity 检查清单

1. **窗口**：背屏使用 `applyRearOpaqueWindowBase()`（或充电动画专用 flags），勿长期透明占位。
2. **输入**：`onCreate` / `onResume` / `onWindowFocusChanged` / 迁屏轮询到背屏时调用 `primeRearSystemBackGestures()`。
3. **返回**：`onCreate` 注册 `OnBackPressedCallback`，用户结束走 `finishProjectionFromUser` → `finish()`。
4. **退出**：不在 `finish` 前 `Session.release`；`onDestroy` 后再 `scheduleOfficialSubscreenRestoreAfterDestroy()`。
5. **沉浸式**：只藏状态栏，保留 `navigationBars` + `BEHAVIOR_DEFAULT`。
6. **官方背屏**：启动链 `finalizeOnRear` 已 `acquire`；Activity `onServiceConnected` 可 `reinforceOfficialSubscreenDisabled`。
7. **迁屏**：主屏占位 + `40ms` 轮询 `displayId == 1` 再建 UI，勿固定秒级盲等。

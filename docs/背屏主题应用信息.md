## REAREye 后屏壁纸管理 / 主题集成方案

本文档整理 REAREye「后屏壁纸管理器」与系统应用（`com.android.thememanager`、`com.xiaomi.subscreencenter`）之间的交互逻辑，说明：

- **如何获取当前壁纸 / 主题信息**
- **如何直接应用某个壁纸**
- **如何读取与保存壁纸模板配置参数**
- **如何在其他项目中复用这一套机制**

术语约定：

- 「壁纸管理器」：REAREye 中的后屏壁纸管理 UI 与逻辑。
- 「Hook Host」：提供实际 IPC 服务的宿主应用，这里是 `com.xiaomi.subscreencenter`。
- 「主题管理器」：系统主题管理应用 `com.android.thememanager`。

---

## 1. 总体架构概览

核心模块及职责：

- `rear-widget-api` 模块  
  - `IRearWallpaperApiService`：AIDL 定义的跨进程服务接口。
  - `RearWallpaperApiClient`：客户端封装，负责绑定远程服务并发起 IPC 调用。
  - `RearWallpaperApiContract` / `RearWidgetApiContract`：约定 action、权限、Bundle key 等常量。

- App 内部仓库层  
  - `RearWallpaperRepository`：挂在 App 进程内，使用 `RearWallpaperApiClient` 对外提供挂起函数：
    - 加载壁纸目录 / 当前壁纸
    - 切换壁纸
    - 导入 / 更新壁纸包
    - 生成预览、删除壁纸
    - 模板配置读取与保存
    - 定时切换排程同步

- Hook 到系统主题管理器  
  - `ThemeManagerScope` + `RearWallpaperThemeManagerSyncHook`  
    - 使用 DexKit 定位 `com.android.thememanager` 里的 `RearListDataManager`、`RearScreenListItemBean` 等类。
    - Hook 其过滤/组装列表的方法，在返回列表中合并 REAREye 导入的壁纸项，使其在系统主题管理器中也可见与可用。
    - 运行时数据来源：`/data/system/theme_magic/users/<userId>/rearScreen/runtime.json`。

整体调用链：

1. UI 调用 `RearWallpaperRepository.loadCatalog` 获取**所有可用后屏壁纸 + 当前选中项**。
2. 需要应用时调用 `RearWallpaperRepository.switchWallpaper`。
3. 若壁纸支持模板配置，UI 调 `resolveTemplateConfigState` 读取配置，再调 `saveTemplateConfig` 回写。
4. REAREye 导入的壁纸会被写入运行时 JSON，由 `RearWallpaperThemeManagerSyncHook` 在主题管理器侧合并进入列表。

## 2. AIDL 服务接口：如何“直接应用”和“配置参数”

`rear-widget-api/build/generated/aidl.../IRearWallpaperApiService.java`（由 AIDL 自动生成）定义了对外提供的方法，包括：

- **目录与预览**
  - `Bundle getCatalog()`：获取完整壁纸目录及当前状态。
  - `byte[] getPreview(int wallpaperId)`：按 ID 获取预览图片字节。

- **直接应用壁纸**
  - `boolean switchWallpaper(int wallpaperId)`  
    - 由宿主服务根据 `wallpaperId` 切换实际后屏壁纸。

- **定时排程**
  - `boolean syncSchedule(boolean enabled, String scheduleData)`  
    - 同步开启状态与序列化后的排程数据。

- **导入 / 更新壁纸包**
  - `Bundle importWallpaperPackage(ParcelFileDescriptor packageFd, String displayNameHint, String previewUri, Bundle options)`
  - `Bundle updateWallpaperMetadata(int wallpaperId, String previewUri, Bundle options)`
  - `Bundle updateWallpaperPackage(int wallpaperId, ParcelFileDescriptor packageFd, String displayNameHint, String previewUri, Bundle options)`
  - `Bundle generateWallpaperPreview(int wallpaperId)`
  - `Bundle deleteWallpaper(int wallpaperId)`

- **模板配置（主题参数）**
  - `Bundle resolveTemplateConfigState(int wallpaperId, String currentOneConfigJson)`
  - `Bundle saveTemplateConfig(int wallpaperId, String oneConfigJson)`

其中与「直接应用 / 主题参数配置」紧密相关的是：

- **直接应用**：`switchWallpaper(int wallpaperId)`
- **读取当前配置状态**：`resolveTemplateConfigState(int wallpaperId, String currentOneConfigJson)`
- **保存配置**：`saveTemplateConfig(int wallpaperId, String oneConfigJson)`

返回的 `Bundle` 结果统一通过 `RearWallpaperApiContract.BundleKeys` 中的 key 约定：

- 成功标记：`SUCCESS`
- 错误信息：`ERROR`
- 可能附带 `WALLPAPER_ID` 等辅助字段。

## 3. RearWallpaperApiClient：客户端如何调用服务

文件：`rear-widget-api/src/main/java/hk/uwu/reareye/widgetapi/RearWallpaperApiClient.kt`

关键点：

- **绑定服务**

  - 通过广播 `RearWallpaperApiContract.ACTION_REQUEST_HOOK_SERVICE` 向 Hook Host（`RearWidgetApiContract.HOOK_HOST_PACKAGE = "com.xiaomi.subscreencenter"`）发送请求，并携带一个 `IRearWallpaperApiConnection` Binder 作为回调。
  - Hook Host 收到广播后启动/绑定真正的 `IRearWallpaperApiService`，并通过回调把 `service` 传回客户端。

  - 相关常量：
    - 权限：`RearWallpaperApiContract.SERVICE_PERMISSION = RearWidgetApiContract.SERVICE_PERMISSION`
    - Action：`RearWallpaperApiContract.ACTION_REQUEST_HOOK_SERVICE`
    - Extra key：`RearWallpaperApiContract.Extras.BINDER`、`Extras.BUNDLE`、`Extras.FORCE_SYNC`

- **方法封装**

  - `getCatalog()` / `getPreview()` / `switchWallpaper()` / `syncSchedule()` 直接转调远端接口。
  - 导入/更新/预览/删除等接口返回 `Bundle`，如果远端返回 `null`，会构造一个失败的 `Bundle`：

    - `SUCCESS = false`
    - `ERROR = "hook service returned empty ... result"`

  - 模板配置相关：

    - `resolveTemplateConfigState(wallpaperId, currentOneConfigJson)`：封装为 `RearWidgetTemplateConfigState` 模型。
    - `saveTemplateConfig(wallpaperId, oneConfigJson)`：直接返回 `Bundle`，需由上层解析 `SUCCESS` / `ERROR`。

## 4. RearWallpaperRepository：App 侧“壁纸管理器”的接口

文件：`app/src/main/java/hk/uwu/reareye/repository/rearwallpaper/RearWallpaperRepository.kt`

### 4.1 获取当前应用壁纸 / 主题状态

**核心函数：**

- `suspend fun loadCatalog(context: Context): RearWallpaperCatalog`

逻辑：

1. 调用 `withRemote(context) { client -> client.getCatalog() }` 获取远端返回的 `Bundle`。
2. 从 `Bundle` 读取：
   - `CURRENT_INDEX`（当前选中在列表中的索引）
   - `CURRENT_WALLPAPER_ID`（当前壁纸的 ID）
3. 使用 `parseCatalogItems(bundle)` 把 `ITEMS` 数组映射为内部模型 `RearWallpaperInfo` 列表，并同步本地预览缓存。

因此，**“当前应用的主题/壁纸”** 实际上通过 `getCatalog()` 返回的 `Bundle` 中的：

- `RearWallpaperApiContract.BundleKeys.CURRENT_INDEX`
- `RearWallpaperApiContract.BundleKeys.CURRENT_WALLPAPER_ID`

UI 层只需持有 `RearWallpaperCatalog`：

- `wallpapers: List<RearWallpaperInfo>`
- `currentIndex: Int`
- `currentWallpaperId: Int?`

就可以高亮当前使用的壁纸，并在列表中标记。

### 4.2 直接应用某个壁纸

- 函数：`suspend fun switchWallpaper(context: Context, wallpaperId: Int): Boolean`

实现：

- `withRemote(context) { client -> client.switchWallpaper(wallpaperId) }`

调用链最终到达 `IRearWallpaperApiService.switchWallpaper`，由 Hook Host 控制实际切换逻辑。

**在 UI 中的使用模式：**

1. 用户在列表中选择某一项（拿到其 `wallpaperId`）。
2. 调用 `RearWallpaperRepository.switchWallpaper(context, wallpaperId)`。
3. 根据返回的 `Boolean` 决定提示成功或错误。

### 4.3 模板配置（主题参数）的读取与保存

相关方法：

- 读取当前模板状态：

  - `suspend fun resolveTemplateConfigState(context, wallpaperId, currentOneConfigJson: String?): RearWidgetTemplateConfigState?`
  - 内部调用：`client.resolveTemplateConfigState(wallpaperId, currentOneConfigJson)`

- 保存模板配置：

  - `suspend fun saveTemplateConfig(context, wallpaperId, oneConfigJson: String?): RearWallpaperOperationResult`
  - 内部调用：`client.saveTemplateConfig(wallpaperId, oneConfigJson)`
  - 结果通过 `parseOperationResult` 解析 `Bundle`，转换成 `RearWallpaperOperationResult(success, wallpaperId?, error?)`。

**典型使用流程：**

1. UI 根据 `wallpaper.templateConfigAvailable` 判断某个壁纸是否支持配置。
2. 打开配置界面时：
   - 可传入上次保存的 `oneConfigJson`，调用 `resolveTemplateConfigState` 获取规范化后的状态与默认值。
3. 用户修改配置后：
   - 将配置序列化为 `oneConfigJson`，调用 `saveTemplateConfig`。
   - 根据返回的 `RearWallpaperOperationResult.success` + `error` 提示用户。

### 4.4 导入 / 更新壁纸时配置参数的传递

导入壁纸有两种形态：

- 从 Uri 导入：`importWallpaperPackage(...)`
- 从内存字节导入：`importWallpaperBytes(...)`

两者最终都会构建一个 `.mrc` 包，并通过 `ParcelFileDescriptor` 交给 `client.importWallpaperPackage(...)`。  
元数据配置（作者、设计师、标题、副标题、子类型、是否第三方、是否支持 AON 等）通过：

- `RearWallpaperMetadataOptions`
- 在 `toMetadataJsonBytes()` 中转为 JSON 写入 `metadata.mrm`。
- 或在 `toBundle()` 中放进 `Bundle`，通过 `RearWallpaperApiContract.BundleKeys.META_*` 传到服务端。

这样做的目的，是保证：

- Hook Host 能在导入时完整获得与主题相关的参数，进而在系统主题管理器的 UI 中正确展示作者、描述、分类、是否可编辑等属性。

## 5. Hook 主题管理器：导入壁纸如何出现在系统列表

文件：`app/src/main/java/hk/uwu/reareye/hook/scopes/thememanager/ThemeManagerScope.kt`  
文件：`app/src/main/java/hk/uwu/reareye/hook/scopes/thememanager/modules/RearWallpaperThemeManagerSyncHook.kt`

### 5.1 Scope 注册

`ThemeManagerScope` 仅负责把多个 Hook 模块挂到一起：

- `UnlockVideoRestrictionsHook`
- `UnlockTemplateMaximumLimitHook`
- `UnmuteVideoWallpaperHook`
- `RearWallpaperThemeManagerSyncHook`（与后屏壁纸同步相关）

### 5.2 RearWallpaperThemeManagerSyncHook 关键逻辑

`onHook()` 中：

1. `loadApp("com.android.thememanager") { ... }`：只在主题管理器进程中生效。
2. 使用 DexKit 检索：
   - `RearListDataManager` 类名
   - 其过滤方法（返回 `List`，内部包含 `getRearListFromDB invalidMtzList` 字符串）
   - `RearScreenListItemBean` 类名（包含 `convertToResource` 方法）
3. Hook 过滤方法，在 `after` 回调中：
   - 拿到原始列表 `original: List<*>`
   - 调用 `mergeImportedWallpapers(original, itemBeanClassName)` 返回新的列表作为 `result`。

`mergeImportedWallpapers` 的主要工作：

1. 从运行时文件读取导入记录：

   - 路径：`/data/system/theme_magic/users/{currentUserId}/rearScreen/runtime.json`
   - 其中包含由 REAREye 导入的壁纸条目（通过 `resId` 前缀 `reareye_import_` 或路径中是否含有 `/reareye_import_` 判断）。

2. 将每条 `RuntimeRecord` 转化为 `RearScreenListItemBean`：

   - 利用反射构造器：`itemBeanClassName.toClass().resolve().firstConstructor { parameterCount = 31 }`
   - 参数中包含：
     - `resType`、`resId`、`resSubType`、`resTypeName`、`applyId` 等基础字段。
     - `resLocalPath`、`metaPath`、`snapshotPreviewPath` 等资源路径。
     - `editable`、`isThirdParties`、`supportAon` 等能力标记。

3. 与原列表去重合并：

   - 避免与原生项重复（通过 `getResId()` + `getApplyId()` 组合 key 判断）。
   - 导入项插在前面，原有项依次追加。

**效果：**

- REAREye 导入的后屏壁纸，就像系统自带壁纸一样出现在 `com.android.thememanager` 的后屏壁纸列表中，并携带完整的元数据与配置能力。

## 6. 在其他项目中移植的步骤

下面是一个**抽象化的移植 Checklist**，你可以根据实际包名 / 模块名调整。

### 6.1 复用 / 搭建 API 层

1. **复制 `rear-widget-api` 模块** 或者在新项目中新建等价模块，包含：
   - AIDL 接口：`IRearWallpaperApiService.aidl`、`IRearWallpaperApiConnection.aidl`。
   - 客户端实现：`RearWallpaperApiClient`。
   - 协议常量：`RearWallpaperApiContract`、`RearWidgetApiContract` 与 `RearWidgetApiModels.kt` 中相关数据类。
2. 根据你的 Hook Host 包名调整：
   - `RearWidgetApiContract.HOOK_HOST_PACKAGE`
   - `RearWidgetApiContract.SERVICE_PERMISSION`
3. 在 Hook Host 侧（如一个独立模块 / Xposed 模块）实现实际的 `IRearWallpaperApiService` 服务逻辑：
   - 响应 `ACTION_REQUEST_HOOK_SERVICE` 广播，获取 `binder` 回调。
   - 通过回调传回 `IRearWallpaperApiService` 实例。

### 6.2 在 App 内实现仓库层

1. 创建类似 `RearWallpaperRepository` 的单例对象或类：
   - 统一封装对 `RearWallpaperApiClient` 的调用。
   - 提供挂起函数：
     - `loadCatalog(context)`
     - `switchWallpaper(context, wallpaperId)`
     - `importWallpaperPackage(...)` / `importWallpaperBytes(...)`
     - `updateWallpaperMetadata(...)` / `updateWallpaperBytes(...)`
     - `generateWallpaperPreview(...)` / `deleteWallpaper(...)`
     - `resolveTemplateConfigState(...)` / `saveTemplateConfig(...)`
     - `syncSchedule(...)`
2. 尽量保持方法签名与当前实现一致，便于直接复用 UI 层代码。

### 6.3 UI 层使用方式

- **获取当前壁纸 / 主题：**
  - 调 `loadCatalog(context)`，从返回的 `RearWallpaperCatalog` 中读取：
    - `currentWallpaperId`
    - `currentIndex`
    - `wallpapers` 列表（`RearWallpaperInfo` 包含是否支持模板配置、是否可编辑、是否第三方等）。

- **直接应用壁纸：**
  - 调 `switchWallpaper(context, wallpaperId)` 即可。

- **配置主题参数（模板配置）：**
  - 加载时调 `resolveTemplateConfigState(...)`，生成初始配置 UI。
  - 保存时调 `saveTemplateConfig(...)`，根据 `success` / `error` 提示用户。

### 6.4 集成到系统主题管理器（可选）

如果你也需要让导入的后屏壁纸出现在系统主题管理器中，可以参考：

1. Hook 目标应用（例如 `com.android.thememanager`）。
2. 使用类似 `RearWallpaperThemeManagerSyncHook` 的 DexKit + 反射逻辑：
   - 定位壁纸列表管理类与其构建/过滤列表的方法。
   - 从你自己的运行时数据文件中读取导入壁纸元数据（可以沿用 `/data/system/theme_magic/users/<userId>/rearScreen/runtime.json` 的结构，也可以自定义）。
   - 反射构造 `RearScreenListItemBean`（或目标系统的等价类）并合入原列表。

这一步需要针对目标 ROM / 主题管理器实现做适配，但整体模式可以直接沿用本项目的做法。

## 7. 总结

- **获取当前应用的壁纸/主题状态**：依赖 `IRearWallpaperApiService.getCatalog()` 返回的 `Bundle`，由 `RearWallpaperRepository.loadCatalog` 解包为 `RearWallpaperCatalog`。
- **直接应用壁纸**：通过 `RearWallpaperRepository.switchWallpaper` → `RearWallpaperApiClient.switchWallpaper` → `IRearWallpaperApiService.switchWallpaper` 调用链实现。
- **配置主题参数（模板配置）**：通过 `resolveTemplateConfigState` 读取当前配置状态，`saveTemplateConfig` 写回，元数据通过 `RearWallpaperMetadataOptions` 和 `RearWallpaperApiContract.BundleKeys.META_*` 传递。
- **与系统主题管理器联动**：`RearWallpaperThemeManagerSyncHook` 在 `com.android.thememanager` 侧 Hook 列表构建方法，将 REAREye 导入的壁纸合并到系统列表中。

将上述模块（AIDL + ApiClient + Repository + Hook 逻辑）拆分/复制到新项目，即可复用一整套「壁纸管理器 + 主题参数配置 + 系统主题联动」的能力。
